Реализация собственной файловой системы как модуля ядра поверх блочного
устройства.

## Содержимое проекта

| Файл              | Назначение                                                 |
|-------------------|------------------------------------------------------------|
| `selfs.h`          | общие структуры (on-disk суперблок, in-RAM `sb_info` и т.д.)|
| `selfs_ioctl.h`    | UAPI-заголовок IOCTL, общий для ядра и userspace           |
| `selfs.c`          | сам модуль ядра                                            |
| `test_selfs.cpp`   | userspace-утилита (C++17)                                  |
| `Makefile`        | сборка модуля и утилиты                                    |
| `README.md`       | этот файл                                                  |

## Сборка

```bash
make
make module
make test
make clean
```

По умолчанию модуль собирается под текущее ядро
(`/lib/modules/$(uname -r)/build`). Чтобы собрать под другое дерево исходников:

```bash
make KDIR=/usr/src/linux-6.12.0
```

Для сборки потребуются: пакет `linux-headers-$(uname -r)` (или собранное
дерево ядра 6.12.x), `gcc`, `g++` с поддержкой C++17, `make`.


## Параметры модуля

| Параметр            | Тип     | По умолчанию | Назначение                                                              |
|---------------------|---------|--------------|-------------------------------------------------------------------------|
| `device`            | string  | (пусто)      | Имя блочного устройства, на котором при загрузке модуля будет           |
|                     |         |              | принудительно создана ФС (форматирование). Если пусто — модуль          |
|                     |         |              | просто регистрируется в VFS; форматирование произойдёт автоматически    |
|                     |         |              | при первом `mount`, если на устройстве нет валидного суперблока.        |
| `sb_first_offset`   | uint    | 0            | Сектор размещения **первой** копии суперблока.                          |
| `sb_second_offset`  | uint    | 1024         | Сектор размещения **второй** копии суперблока (должен быть `>` первого).|
| `max_name_len`      | uint    | 32           | Максимальная длина имени файла (без учёта `\0`); потолок — 63.          |
| `max_file_sectors`  | uint    | 4            | Максимальный размер файла в секторах (M).                               |

Сектор всегда 512 байт. Имена файлов генерируются автоматически вида
`f00000`, `f00001`, …


## Раскладка на диске

```
сектор 0  ┌────────────────────────────┐  ← sb_first_offset
          │      SuperBlock (CRC32)    │
          ├────────────────────────────┤  ← регион файлов #1
          │  файл #0  (M секторов)     │
          │  файл #1  (M секторов)     │
          │             …              │
          ├────────────────────────────┤  ← sb_second_offset
          │   Duplicate SuperBlock     │
          ├────────────────────────────┤  ← регион файлов #2
          │  файл #K  (M секторов)     │
          │             …              │
          └────────────────────────────┘  ← bdev_nr_sectors
```

Каждый файл размещается **строго** на `M` подряд идущих секторах.
Метаданные файлов (имя, смещение, размер) **детерминированно**
вычисляются из параметров суперблока — на диске их хранить не нужно.


## Проверка целостности суперблока

Каждый суперблок содержит поле `hash` — CRC32 по всем 512 байтам блока
с обнулённым полем `hash`. При монтировании:

1. Читается первая копия. Если магия совпадает, версия совпадает и CRC
   валиден — используется она.
2. Если первая копия повреждена — читается вторая. Если она валидна —
   используется она, **а первая копия восстанавливается из второй**.
3. Если обе копии невалидны — диск форматируется заново (две свежие
   копии записываются на свои места).

## Пример использования

Удобнее всего работать через loop-устройство:

```bash
# 1. Создаём образ диска (10 МиБ)
dd if=/dev/zero of=disk.img bs=1M count=10

# 2. Подключаем как loop-устройство
sudo losetup /dev/loop0 disk.img

# 3. Грузим модуль. device=/dev/loop0 — отформатирует устройство
#    сразу при загрузке. Если опустить device, форматирование произойдёт
#    при первом mount.
sudo insmod selfs.ko                 \
        device=/dev/loop0           \
        sb_first_offset=0           \
        sb_second_offset=1024       \
        max_name_len=32             \
        max_file_sectors=4

# 4. Монтируем
sudo mkdir -p /mnt/selfs
sudo mount -t selfs /dev/loop0 /mnt/selfs

# 5. Смотрим, какие файлы создались
ls /mnt/selfs | head

# 6. Гоняем userspace-тест: пишет в каждый файл случайное число
#    и читает его обратно
sudo ./test_selfs test /mnt/selfs

# 7. Метаданные + CRC32 каждого файла
sudo ./test_selfs meta /mnt/selfs

# 8. Маппинг секторов для конкретного файла
sudo ./test_selfs sectors /mnt/selfs f00000

# 9. Обнулить все файлы
sudo ./test_selfs zero /mnt/selfs

# 10. Стереть всю ФС (обнулить файлы + обе копии суперблока)
sudo ./test_selfs erase /mnt/selfs

# 11. Размонтирование и выгрузка
sudo umount /mnt/selfs
sudo rmmod selfs
sudo losetup -d /dev/loop0
```

После `erase` суперблоки на диске уничтожены; следующая попытка
монтирования приведёт к автоматическому переформатированию.

## IOCTL

UAPI описан в `selfs_ioctl.h`. IOCTL можно выдать **либо** на сам
mountpoint (директория `/mnt/selfs`), **либо** на любой файл внутри —
обработчик подключён и к `dir_ops`, и к `file_ops`.

| IOCTL                    | Назначение                                                                                |
|--------------------------|-------------------------------------------------------------------------------------------|
| `SELFS_IOCTL_ZERO_ALL`    | Обнуляет содержимое всех файлов (`_IO('M', 1)`).                                          |
| `SELFS_IOCTL_ERASE_FS`    | Обнуляет файлы и **обе копии суперблока** (`_IO('M', 2)`). После этого требуется umount.  |
| `SELFS_IOCTL_GET_META`    | Возвращает `struct selfs_meta_list` со списком файлов и CRC32 содержимого (`_IOR('M', 3)`).|
| `SELFS_IOCTL_GET_SECTORS` | По имени файла возвращает `struct selfs_sector_map` со списком занимаемых секторов на диске (`_IOWR('M', 4)`). |

Все размеры (количества файлов / секторов) ограничены константами
`SELFS_MAX_FILES_IOCTL` и `SELFS_MAX_SECTORS_MAP` из `selfs_ioctl.h`.

## CLI userspace-утилиты

```
test_selfs test    <mountpoint>              # запись + чтение случайного uint32_t в каждый файл
test_selfs zero    <path>                    # ZERO_ALL
test_selfs erase   <path>                    # ERASE_FS (после требуется umount)
test_selfs meta    <path>                    # GET_META: таблица name / offset / size / CRC32
test_selfs sectors <path> <filename>         # GET_SECTORS: список секторов файла на диске
```

`<path>` — точка монтирования или любой файл внутри неё.
Для всех команд, кроме `test`, потребуются права root (а для `test` —
доступ на запись в `<mountpoint>`, что обычно означает то же самое).
